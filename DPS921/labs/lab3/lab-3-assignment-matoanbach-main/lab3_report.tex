\section{Question 1: Sequential Matrix Multiplication}
\subsection{Implementation Summary}
The CPU version uses the standard three-loop matrix multiplication algorithm. For each output element \(C[i,j]\), the program computes the dot product of row \(i\) from matrix \(A\) and column \(j\) from matrix \(B\). The implementation accepts the matrix size as a command-line argument and reports the average execution time over five runs.

\subsection{Core CPU Algorithm}
\begin{lstlisting}[style=cppstyle,language=C++]
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        float sum = 0.0f;
        for (int k = 0; k < N; k++) {
            sum += A[i * N + k] * B[k * N + j];
        }
        C[i * N + j] = sum;
    }
}
\end{lstlisting}

\section{Question 2: CUDA Matrix Multiplication}
\subsection{Part A: Basic CUDA Implementation}
The first CUDA version uses global memory only. Each thread computes exactly one output element of matrix \(C\). A two-dimensional block and grid configuration is used so that thread coordinates map naturally to matrix row and column positions.

\subsection{Basic CUDA Kernel}
\begin{lstlisting}[style=cppstyle,language=C++]
__global__ void matrixMultiplyBasicCUDA(float *A, float *B, float *C, int N)
{
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    if (row < N && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < N; k++) {
            sum += A[row * N + k] * B[k * N + col];
        }
        C[row * N + col] = sum;
    }
}
\end{lstlisting}

\subsection{Part B: Tiled Shared-Memory CUDA Implementation}
The optimized CUDA version uses shared memory tiling. A tile size of 16 $\times$ 16 was selected because it matches the block configuration required by the assignment and provides a practical balance between occupancy and shared-memory reuse on common GPUs.

Shared memory improves performance because threads in the same block can reuse data loaded from global memory. This reduces repeated global-memory reads and lowers overall memory access latency.

\subsection{Tiled CUDA Kernel Summary}
\begin{itemize}
\item Two shared tiles are declared: one for a sub-block of \(A\) and one for a sub-block of \(B\).
\item Each thread cooperatively loads one value from each matrix tile.
\item Threads synchronize before using the tile values.
\item Each thread accumulates a partial sum across all required tiles.
\item The final result is written to the corresponding output element in global memory.
\end{itemize}

\subsection{Performance Discussion}
Answer the required discussion points below.

\begin{enumerate}
\item Which GPU implementation performed better, the basic version or the tiled shared-memory version?
\item How much speedup did each GPU implementation achieve relative to the CPU implementation?
\item Why does shared memory improve performance for matrix multiplication?
\item What overheads are included in the total CUDA execution time but not in kernel execution time?
\end{enumerate}

\noindent Response:\\
Based on total execution time, the tiled shared-memory implementation performed better than the basic CUDA implementation for all three tested matrix sizes. For 256 $\times$ 256, the basic CUDA version achieved a speedup of about 1.10$\times$ over the CPU, while the tiled version achieved about 22.34$\times$. For 512 $\times$ 512, the basic CUDA version achieved about 7.31$\times$ speedup, while the tiled version achieved about 31.19$\times$. For 1024 $\times$ 1024, the basic CUDA version achieved about 35.13$\times$ speedup, while the tiled version achieved about 60.68$\times$. Shared memory improves performance in theory because threads in the same block can reuse matrix values that were loaded once from global memory, which reduces repeated slow global-memory accesses. However, in these measurements, the tiled kernel was not always faster when comparing kernel-only time. This suggests that synchronization cost, tile-loading overhead, cache behavior, or implementation details may have reduced the expected kernel-level benefit. The total CUDA execution time includes overheads that are not part of kernel execution time, such as device memory allocation, host-to-device copies, device-to-host copies, synchronization, and kernel launch overhead. Because the total-time results also include startup and transfer costs, they should be interpreted more cautiously than the kernel-only measurements. The basic total time is noticeably higher for small matrix sizes because the basic kernel is executed first in the program and may include CUDA context initialization overhead. This overhead is not part of the matrix multiplication work itself, but it can appear in the first CUDA API calls such as \texttt{cudaMalloc} or \texttt{cudaMemcpy}. A more controlled benchmark would warm up the CUDA runtime before timing each implementation.

\subsection{Discussion}

\noindent Response:\\
The Thrust-based implementations produced correct results for all tested matrix sizes. Thrust simplified memory management by replacing manual \texttt{cudaMalloc} and \texttt{cudaMemcpy} calls with \texttt{thrust::host\_vector} and \texttt{thrust::device\_vector}, while still allowing the same CUDA kernels to be launched through \texttt{thrust::raw\_pointer\_cast()}. Based on total execution time, the tiled Thrust version performed better than the basic Thrust version for 256 $\times$ 256, 512 $\times$ 512, and 1024 $\times$ 1024. For 2048 $\times$ 2048, however, the basic Thrust version performed better than the tiled Thrust version. Approximate speedups relative to the CPU using total execution time were 1.08$\times$ versus 20.13$\times$ at 256 $\times$ 256, 6.77$\times$ versus 29.36$\times$ at 512 $\times$ 512, 36.50$\times$ versus 61.34$\times$ at 1024 $\times$ 1024, and 127.03$\times$ versus 113.66$\times$ at 2048 $\times$ 2048 for the basic and tiled Thrust versions respectively. Thrust is a good choice when development speed, code clarity, and safer memory management are more important than fully manual control over every CUDA memory operation.

\section{Conclusion}
The sequential CPU implementation produced correct matrix multiplication results and provided a baseline for comparison. Both CUDA implementations in Question 2 also produced correct results for all tested sizes. In these measurements, the tiled shared-memory version achieved better total execution time than the basic CUDA version, even though the basic version sometimes had lower kernel-only time. This shows that total performance depends not only on kernel speed, but also on memory transfers, allocation overhead, and synchronization. The optional Thrust implementation produced correct results as well and demonstrated that higher-level GPU memory management can still provide strong performance while making the code easier to write and maintain.
