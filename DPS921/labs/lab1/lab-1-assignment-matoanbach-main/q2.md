## Theoretical Speedup Estimate

Method: Amdahl's Law.

Definitions:

- `T1`: time with 1 thread
- `Tp`: time with `p` threads
- `alpha`: fraction of runtime that can be parallelized

Formulas:

- `Speedup(p) = 1 / ((1 - alpha) + alpha / p)`
- `Efficiency(p) = Speedup(p) / p`

Assumptions / how `alpha` was estimated:

- Parallelized section: magnitude loop (`sum += x[i] * x[i]`) using data-parallel chunking + reduction.
- Serial overhead: allocation/initialization, thread management, and final reduction.
- `alpha` estimate source (pick one and fill):
  - From Q1 timing split: `alpha = T_mag / (T_alloc_init + T_mag)`
  - Or: treat magnitude section as the main parallel target and discuss overhead separately.

Computed estimate (fill):

Using Q1 (O2) at `n = 30,000,000`:

- `T_alloc_init = 19.3333 ms`
- `T_mag = 33.3333 ms`
- `alpha = T_mag / (T_alloc_init + T_mag) = 0.6329`

Amdahl speedup estimates:

- `Speedup(2) = 1.4630`
- `Speedup(4) = 1.9036`
- `Speedup(8) = 2.2411`

Note: these Amdahl estimates are based on *total* runtime (including allocation/init). In the Results section below, speedup is computed from the magnitude timer only, so observed speedup can be higher than this estimate.

## Decomposition Pattern Used

- Primary pattern: **Geometric Decomposition (1D data partitioning)**
  - The input array indices `[0, n)` are split into `p` contiguous chunks.
- Execution structure: **Fork/Join**
  - Fork: create `p` threads.
  - Join: wait for all threads to finish.
- Computation structure: **Map-Reduce**
  - Map: each thread computes a partial sum of squares over its chunk.
  - Reduce: main thread sums partial results and applies `sqrt`.

Justification:

- Each element is processed independently with the same operation, so data-parallel chunking minimizes synchronization.
- A single reduction at the end is simple and has low overhead relative to the main loop for large `n`.

## Implementation Summary

Program: `q2.cpp`

- Command line: `./q2 <n> <numThreads>`
- Each thread computes:
  - `partial[t] = sum_{i=start..end-1} x[i]^2`
- Final result:
  - `sqrt(sum(partial))`

## Comparison: Theoretical vs Observed

- Theoretical speedup (from Amdahl, total runtime estimate at `n=30,000,000`):
  - `S(2)=1.4630`, `S(4)=1.9036`, `S(8)=2.2411`
- Observed speedup (magnitude timer only, `n=30,000,000`):
  - `S(2)=2.2041`, `S(4)=4.1538`, `S(8)=5.4000`

Discussion of differences:

- Amdahl estimate above includes a serial portion (allocation/init) that is not included in the magnitude-only speedup calculation.
- Thread overhead and the final reduction reduce ideal scaling (speedup < p).
- Memory hierarchy effects can change per-thread performance (cache behavior, bandwidth contention), especially as `p` increases.
- Diminishing returns appear at higher thread counts as overheads and bandwidth limits become more significant.
